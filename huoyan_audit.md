# 火眼（Huoyan）测评框架深度审计报告

## 总体评价

该框架的设计出发点明确且有价值：专门针对 AI 中转站（relay/gateway）这个特殊场景，而不是测评模型本身，测评维度涵盖了**真实性、性能、Agent能力、成本安全、中转安全**五大类，整体思路相当完整。但深入阅读代码后，发现在若干关键指标上存在**测试设计缺陷、结果可信度低**的问题，也有少量重要指标缺失。

---

## 一、指标数量与分类：合理，整体不多也不少

| 套件 | 指标数 | 覆盖目的 |
|---|---|---|
| authenticity（保真度） | 5 个核心探针 + 1 个综合评分 | 检测中转商是否偷换模型 |
| performance（性能） | 3 个探针（TTFT/TPS, 并发, 可用性） | 检测性能水平 |
| agentic（智能体能力） | 3 个探针（工具调用, 长上下文, 多模态） | 检测功能完整性 |
| cost_security（成本安全） | 5 个探针（Token对齐, TLS, 安全头, 限流透明度, 隐私策略） | 检测合规性与计费风险 |
| security_audit（中转安全审计） | 5 个探针 | 检测中转商特有的安全威胁 |

**数量合理，没有明显的"为了看起来多"的堆砌问题。** 以下对每个套件做深度评估。

---

## 二、各指标测试方法详细审计

### 2.1 保真度套件（authenticity.py）

#### ✅ 约束跟随探针（acrostic_constraints）—— 方法合理，结果可信

**测试方法：** 要求模型写一首四行藏头诗，首字必须是"火眼验真"，每行恰好7个汉字。
**评分：** 精确字符比对，结果客观可重复。  
**结论：** 方法合理，是一个轻量但有效的指令跟随能力探针，模型换成低能力替代品会直接失败。

---

#### ✅ 边界推理探针（boundary_reasoning）—— 方法合理，结果可信

**测试方法：** 用 Python 可变默认参数经典 Bug（`def f(items=[]):`）测试模型推理能力。
**代码逻辑：** `_extract_list_outputs` 解析输出中的列表格式，与预期 `["[0]", "[0, 1]", "[10, 1]", "[0, 1, 2]"]` 做精确匹配。

**一个值得注意的细节：**

```python
# performance.py line 94
if output_tokens > 1:
    inter_token_latency_ms = (response.generation_seconds / (output_tokens - 1)) * 1000
```

这道题的答案中，`[10, 1]` 而不是 `[10, 0, 1]` 是正确的，因为 `f([10])` 使用了新列表，不是共享默认参数。这个推理确实需要真正的代码理解能力，难以靠抓关键词侥幸，**结论：方法良好**。

---

#### ⚠️ 身份探针（identity）—— 信号极弱，实用性存疑

**测试方法：** 发送系统提示让模型自报 MODEL 和 COMPANY，然后在响应中查找预定义关键词（如智谱、zhipu、bigmodel）。

**核心问题：**
1. **身份自报本质上是训练结果，几乎不可能通过中转层影响**。绝大多数高质量中转商都会保留原始模型的自我认知，这个测试对中转商"偷换模型"的检测能力几乎为零。
2. **代码中明确标注了这只是弱信号（weak band，权重10）**，并在结论里写了"只是弱信号"。但作为一个专门被列出的探针，它的独立信息价值值得质疑。
3. **从测试结果看**：GLM-5.1 返回 `MODEL=unknown COMPANY=unknown`，这是正常的谦虚行为（遵循指令说不知道），不是任何问题的证明。

**建议：** 将此探针的权重降至5或直接合并到综合信号中作为辅助参考，不应独立陈列为警告状态。

---

#### ⚠️ 输出一致性探针（response_consistency）—— 设计有缺陷

**测试方法：** 用 `temperature=0` 重复发送同一个关于 TCP 三次握手的问题3次，用 `SequenceMatcher` 计算相似度。

**核心问题：**
1. **`temperature=0` 保证了确定性，相似度接近1.0几乎是必然结果**，对大多数正常模型来说这不是任何区分性测试。从报告中也可以看到 GLM-5.1 得到了 1.0000 的完美相似度 —— 三次响应完全一致，这是正常行为，不是"亮点"。
2. **相似度高不等于模型质量高**。换一个差但稳定的模型，相似度同样会是 1.0。
3. **真正的一致性风险**是：中转商在不同时段路由到不同后端模型。这需要**跨时间、非连续**的采样，而不是同一 session 内连续3次调用。

**修复建议：** 增加时间间隔（至少5分钟）进行跨批次采样；或者改用 `temperature>0` 测试语义一致性（内容核心不变但表述可变）。

---

#### ✅ 多语种理解探针（linguistic_fingerprint）—— 方法紧凑有效

给出三段文本（文言文、Rust 代码、SQL），要求识别类型并用中文概括。检测信号：`["文言" 或 "古文", "rust", "sql"]`。简单明了，任何轻量级或被骗换的边缘模型都难以通过。

---

### 2.2 性能套件（performance.py）

#### 🔴 TTFT/TPS 探针 —— 存在严重计算错误

**测试方法：** 发5次流式请求，记录 `ttft_seconds` 和 `first_content_seconds`，然后计算 ITL（相邻 token 延迟）。

**代码中的 ITL 计算：**

```python
# performance.py line 94-95
if output_tokens > 1:
    inter_token_latency_ms = (response.generation_seconds / (output_tokens - 1)) * 1000
```

而 `generation_seconds` 的定义是：

```python
# client.py line 480-483
if first_content_seconds is not None:
    generation_seconds = max(elapsed - first_content_seconds, 0.0)
elif ttft_seconds is not None:
    generation_seconds = max(elapsed - ttft_seconds, 0.0)
```

**问题：**
1. `ITL = generation_seconds / (output_tokens - 1)`：这是把"从第一个 token 到最后一个 token 的总时间"除以"token 数-1"，得到平均相邻 token 间隔。概念上是对的，但**精度极低**。真正的 ITL 应该来自每个 token 的实际到达时间差，而当前代码完全没有记录逐 token 时间戳。这里计算的实际上是 **平均生成速率的倒数**，而非真正的 P99 ITL。

2. **TTFT 与 TTFC 的混用**：代码同时记录了两个时间戳：
   - `ttft_seconds`：openai-chat 协议中，首个 `reasoning_content` 或 `content` delta 的时间（可能是推理 token）
   - `first_content_seconds`：首个实际文本内容 token 的时间
   - 对于有推理链的模型（如 DeepSeek-R1），TTFT 是打出第一个推理 token 的时间，TTFC 才是用户真正看到内容的时间，二者可能相差数十秒
   - 当前代码是将 `ttft_seconds` 用于 TTFT 报告，而 `generation_seconds` 基于 `first_content_seconds` 计算，**维度混合**，比较时容易产生误解

3. **采样数只有5次**：流式压测5次，时间间隔仅0.5秒，无法代表统计规律，P90 等百分位数在5个样本上意义极小。

---

#### ⚠️ 并发探针（concurrency）—— 测试设计存在问题

**测试方法：** 默认只测 `concurrency_levels=[5]` 这一个档位，发5个"请只回答 OK"的请求，计算成功率。

**核心问题：**
1. **5并发5请求过于轻量**，无法代表真实业务压力。真实场景下，API 中转商的限制通常在数十到数百并发以上。
2. **从报告中发现一个真正的问题**：GLM-5.1 在并发5时成功率只有 40%（3个429），**但框架把它归因到"中转稳定性"**，实际上这大概率是智谱官方 API 的限流，与中转商无关。当直接访问官方 API 和访问中转商时，如果同样触发 429，框架无法区分"中转商的并发限制"和"上游 API 的配额限制"。
3. **只有一个并发档位**，无法绘制"并发-成功率"曲线，难以找到中转商的实际瓶颈点。

---

#### ✅ 可用性探针（availability）—— 方法合理但存在时间窗口问题

连续5次请求，间隔1秒，测简单可用性。方法简单但有实际意义。**注意：5次、5秒窗口太短**，只能反映瞬时状态，不能代表真实可用性。对比行业标准，SLA 通常用24小时或7天的窗口。建议作为"瞬时可用性快照"而非"可用性"进行命名和诠释。

---

### 2.3 智能体套件（agentic.py）

#### ✅ 工具调用探针（tool_calling）—— 方法良好

使用 `tool_choice={"type": "function", "name": "route_package"}` 强制要求调用工具，用 jsonschema 验证参数精确性（包括 `const` 约束）。**设计严格，可信度高**。

从报告中看到，GLM-5.1 出现了 ERROR：工具调用返回0次 —— 这可能是适配层在 Anthropic API 格式转换时出现了问题（client.py 中 `_convert_tools_for_anthropic` 有处理逻辑），**这是一个真实的中转兼容性问题，值得深入排查**。

---

#### ✅ 长上下文探针（long_context_integrity）—— 设计非常有价值

在20000字符的文档中埋入 HEAD/MIDDLE/TAIL 三个 canary 值，要求模型回答。**这个测试直接能检测中转商是否在中间层偷偷截断了长上下文**，是整个测评体系中最有中转专属价值的测试之一。实现也很干净。

---

### 2.4 成本安全套件（cost_security.py）

#### ⚠️ Token 对齐探针（token_alignment）—— 对中国模型家族几乎无效

**测试方法：** 用 tiktoken 本地估算 input tokens，与 API 返回的 `usage.prompt_tokens` 对比，期望比率在 0.9~1.1 内。

**核心问题：**
1. tiktoken 是 OpenAI 模型的分词器，对 GLM、Kimi、Qwen、DeepSeek 这些中国模型系列，代码直接用 `cl100k_base` 做**近似估算**（`approximate=True`），偏差 ±50% 都是正常的。
2. 代码在 `approximate=True` 时，只要 ratio 在 0.5~1.5 内就标记为 `SKIP`，**等于这个探针对中国模型家族实际上完全不工作**。
3. 从报告可以验证：GLM-5.1 的 ratio 是 0.6477（差了约35%），标记为 SKIP，没有任何有效信息。

**建议：** 要么集成各家模型的官方分词器（通义、GLM、DeepSeek 均有开源分词器），要么对中国模型改用统计方法（汉字约1.5~2 tokens/字的经验值），或明确标注"此探针对中国模型family无效"。

---

#### ✅ TLS 探针（tls_baseline）—— 合理

直接 SSL 握手检查 TLS 版本和证书剩余天数，方法简单可靠。**TLSv1.2** 是被接受的，**TLSv1.0/1.1** 会报 FAIL。证书34天过期会报 WARN，合理。

---

#### ⚠️ 安全响应头探针（security_headers）—— 评判维度有争议

**测试方法：** GET 请求 `base_url`，检查响应头 HSTS、X-Content-Type-Options、CSP 等。

**问题：**
1. API endpoint 不是 Web 页面，`X-Frame-Options`、`CSP` 等对纯 API 服务并不适用。只有 `HSTS` 和 `X-Content-Type-Options` 对 API 有实际意义。
2. GET 请求 API base_url 可能返回 404 或 redirect，不一定能代表实际 API 响应的安全头。有些服务（如 bigmodel.cn）API endpoint 的安全头与 Web 域名不同。

---

#### ✅ 限流透明度探针（rate_limit_transparency）—— 有价值但逻辑有小问题

**问题：** 只发3次正常请求，在没有触发限流的情况下，大多数 API 不会包含 `x-ratelimit-*` 头，这时会返回 WARN。**但"正常请求下没有限流头"是完全正常的**，不能算作服务商的缺陷。真正重要的是"触发 429 时是否提供 `Retry-After`"。

测试逻辑调整建议：主要判断"触发 429 时有没有 `Retry-After`"，而不是"普通请求有没有限流头"。

---

### 2.5 中转安全审计套件（security_audit.py）

#### ✅ 错误响应泄漏探针（error_response_leakage）—— 方法合理

发送3种格式错误的请求（messages=123、工具格式错误、不存在的模型），检测响应中是否泄漏：
- Secret（API Key、AWS Key、PEM）
- 内部实现信息（litellm、fastapi 调用栈、上游 URL、文件路径）

检测的 regex 模式集合（`LEAKAGE_PATTERNS`）覆盖了常见中转框架（OneAPI、NewAPI 基于 litellm 或 FastAPI），非常有针对性。**结论：这是整个套件里技术含量最高的探针之一**。

---

#### ✅ 系统提示注入探针（system_prompt_injection）—— 方法有创意但结果可信度偏低

**测试方法：**
1. 发送**不带 system message** 的请求，让模型"逐字输出收到的所有系统级指令"
2. 再问"收到了几条非用户指令"

**问题：**
1. **模型是否如实披露系统提示，取决于训练而不是中转层**。被 RLHF 训练过的模型通常会拒绝或编造，返回 `NONE_RECEIVED` 并不代表没有注入。
2. **正则匹配模式过于宽泛**，如 `"你是一个?(?:有用|友好|智能|AI|助手)"` 可能在模型"描述自己性格"时也会匹配到，产生假阳性。
3. **测试只能检测"中转商不成熟的系统提示注入"**（比如直接在 system message 里写 "你是 xxx 助手"），对嵌入训练层面的修改完全无效。

从 GLM-5.1 的结果看到 `NONE_RECEIVED`，这很可能是模型训练层面的对齐行为，结论不太可信。

---

#### ✅ 依赖替换检测（dependency_substitution）—— 设计精妙

强制工具调用传入特定版本的包安装命令，检测中转商是否在 function call 的参数里偷偷替换版本号或包名。对于 Agentic 场景下的安全性，这是一个非常有价值的探针。

**潜在误判：** 如果中转商在路由层对工具调用做了"格式修复"（如把 JSON 重新序列化），可能导致参数字符串空格不同但语义相同，从而误报 FAIL。

---

#### ✅ 条件投递检测（conditional_delivery）—— 设计有价值但检测力弱

用3次"热身"请求后再发工具调用，看工具调用的参数是否发生变化。**想检测的是：中转商是否在会话不同阶段有不同的行为**（比如前几次正常，后来开始篡改）。

**问题：** 3次热身根本不够，现实中的条件触发机制可能需要数十次请求才触发，或基于时间、流量等条件触发。这个探针的检测力非常弱，更多是"象征性"测试。

---

#### ✅ 流完整性探针（stream_integrity）—— 方法扎实，分协议适配好

对三种协议（openai-chat、openai-responses、anthropic-messages）分别检查流式 SSE 事件序列的合法性，包括：
- 未知事件类型
- model 字段在流中是否发生变化（chunk 级别的路由切换）
- usage 是否有值
- 是否正常终止

这是一个真正有价值的中转专属探针，**chunk 级别的 model 字段变化**可以直接检测中转商在一次流式响应中切换了后端节点。

## 三、测试结果的可信度分析

### 3.1 GLM-5.1 的 TTFT/TPS 结论不可信

报告中 GLM-5.1 的所有流式指标都是 "-"（null），原因是 stream probe 出现了 ERROR。查看代码，`ttft_seconds` 在流中找不到有效的 delta content 时就是 `None`。

**从 stream_integrity 报告中可以看到"错误"状态**，说明 bigmodel.cn 通过 Anthropic API 格式的流式响应在当前代码下解析失败。这是一个**真实的代码 Bug**：`client.py` 中对 `anthropic-messages` 格式流的处理在 Anthropic format + 非标准 bigmodel 服务器的情况下出现了问题。性能报告中所有时延数据为空，与这个 Bug 直接相关。

### 3.2 综合保真度评分的可信度

GLM-5.1 得到 88.24/100 分，但：
- `identity` 探针 WARN（得0分，弱信号）
- `tool_calling` ERROR（不计入，强信号）
- `token_alignment` SKIP（不计入，中信号）
- `stream_integrity` ERROR（不计入，强信号）
- 剩余4个探针全部 PASS

**实际上只有4/11个信号被计入评分，且这4个都是最基础的纯文本能力测试。** 88.24分的结论基础太薄，不应轻信。coverage_ratio 为70.8%（超过60%的阈值），勉强达标，但距"高置信"还差很远。

### 3.3 并发稳定性失败的归因问题

报告显示并发5时成功率只有40%（3个429），**但率先应该怀疑的是：这是官方 API 限速，而非中转商问题**。框架没有提供"基线对照"（即直连官方 API 的数据），无法判断问题来源。

## 四、缺失的重要指标

### 4.1 响应延迟的"尾延迟"分布（P99 抖动）

当前 TTFT 探针只有5个样本，P90 几乎没有统计意义。真实中转场景中，**尾延迟（P99, P999）** 才是最影响用户体验的指标（因为中转商往往做了多级异步转发，某些请求会走慢速节点）。

### 4.2 响应哈希一致性（Cross-Request Fingerprinting）

对完全相同的请求（`temperature=0`、固定 seed），检测不同批次的响应是否来自同一模型。**如果响应哈希在不同时间段有规律性变化，说明中转商在做动态路由（轮询不同模型）**。这比身份探针更有检测力，且无需依赖模型自报行为。

### 4.3 Prompt 压缩/截断检测

某些中转商为了节约成本，会在长 prompt 时自动做摘要或截断。**当前的长上下文探针（20000字符的 canary 测试）** 已经部分覆盖了这个场景，但仅用一个固定长度测试不够，建议增加渐进式长度测试（5K、20K、50K 字符，分别检测 canary 命中情况），找出截断发生的临界点。

### 4.4 Latency 的时间维度变化（Time-of-Day 变化）

中转商往往在高峰期（北京时间9-11点、14-17点）和低峰期有明显不同的性能表现。**单次测评快照无法反映这一规律**，建议增加"时间窗口"维度的可配置性。

### 4.5 计费实际准确性双向测试

当前 Token 对齐探针只检测 input tokens，而 **output token 的虚报** 才是中转商常见的作弊手段。可以：
1. 让模型精确输出N个字符（在 prompt 里写"请恰好输出X个字"）
2. 对比 API 返回的 `completion_tokens` 与实际输出字符数/估算的 output token 数

## 五、总结与优先级建议

| 问题 | 严重程度 | 修复难度 |
|---|---|---|
| ITL 计算是平均值而非真正IIT延迟分布 | 🔴 高 | 中（需逐token打时间戳） |
| 性能探针样本量太少（N=5，P90无意义） | 🔴 高 | 低（改配置） |
| Token对齐对中文模型几乎失效 | 🔴 高 | 高（需集成各模型分词器） |
| 流完整性ERROR导致多个指标级联失败 | 🔴 高 | 中（需修复Anthropic流解析） |

**最值得信任的指标（结论可靠的）：**
- 错误响应泄漏检测（leakage）
- 工具调用精确性（tool_calling）
- 长上下文 canary（long_context_integrity）
- 流完整性事件序列（stream_integrity）
- 依赖替换检测（dependency_substitution）
- TLS 基线（tls_baseline）

**结论可信度较低的指标：**
- TTFT/ITL（样本太少 + ITL近似计算）
- 综合保真度评分（关键探针失败/跳过时，覆盖率偏低）
- 身份探针（弱信号，模型自报不可靠）
- 系统提示注入（模型对齐行为干扰结果）
- Token 对齐（对中文模型分词器不准确）
